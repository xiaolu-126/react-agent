# 智能主播推荐系统技术调研报告

> 调研目的：为构建主播推荐 Agent 系统，对大语言模型选型、搜索方案、业务流程形态进行综合调研，为技术决策提供依据。
> 调研日期：2026 年
> 数据来源：各厂商官方定价页面（2026 年 5 月更新）、SuperCLUE 中文大模型测评报告（2025-2026）、第三方评测报告

---

## 一、调研背景

随着直播和短视频行业的快速发展，主播推荐内容的生成需求日益增长。传统的人工撰写方式效率低、难以规模化，AI 技术为主播推荐场景提供了新的解决方案。本项目旨在构建一套 AI 智能主播推荐系统，为了做出合理的技术选型，需要从以下几个维度进行调研：

1. **业务流程形态**：工作流 vs 智能代理，哪种更适合该场景
2. **大语言模型**：不同模型的价格、能力、适用场景和局限性
3. **搜索方案**：知识库搜索和网络搜索的选型对比

---

## 二、业务流程形态调研

### 2.1 调研概述

当前 AI 应用的任务处理主要存在两种架构形态：**工作流（Workflow）** 和 **智能代理（Agent）**。两者的本质区别在于"执行路径由谁决定"——前者由开发者预先定义，后者由 AI 自主判断。

### 2.2 工作流（Workflow）形态

工作流是指将任务按预设步骤线性编排，每一步执行完成后进入下一步，直至流程结束。

```
用户输入
    │
    ▼
┌──────────────────┐
│   知识库查询检索   │
│  （本地主播资料库）  │
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│   网络搜索补全     │
│  （实时动态信息）   │
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│   AI 整合生成     │
│  （汇总→推荐文案）  │
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│     输出结果       │
└──────────────────┘
```

**核心特征：**

| 维度 | 说明 |
|------|------|
| 路径控制 | 开发者硬编码，执行路径固定 |
| 确定性 | 每一步的结果可预期，流程可预测 |
| 复杂度 | 实现简单，逻辑清晰，调试容易 |
| 容错 | 某步出错通常导致流程中断 |
| 适用场景 | 流程固定、输入输出明确的成熟业务 |

**典型场景：** 数据 ETL 管道、文档处理流水线、规则明确的客服流程。

### 2.3 智能代理（Agent）形态

智能代理以 ReAct（Reasoning + Acting）为代表，采用"思考→行动→观察"循环机制，AI 自主判断每步操作，直至任务完成。代表框架包括 LangGraph、AutoGen、CrewAI 等。

```
用户输入
    │
    ▼
┌──────────────────────────┐
│      思考（Reason）        │
│  AI 理解需求，决定下一步    │
└────────┬─────────────────┘
         │
    ┌────▼────┐
    │ 决策判断  │
    │──────────│
    │调用工具？│──▶ 行动：执行搜索等
    │直接输出？│──▶ 生成最终结果
    └────┬────┘
         │
    ┌────▼────┐
    │ 观察（Observe）│
    │  分析工具结果   │── 回到思考 ──
    └──────────────┘
```

**核心特征：**

| 维度 | 说明 |
|------|------|
| 路径控制 | AI 自主判断，路径动态调整 |
| 灵活性 | 可根据中间结果调整后续策略 |
| 复杂度 | 实现较复杂，需处理循环和图状态管理 |
| 容错 | 可自我纠错，信息不足时自动补充 |
| 适用场景 | 不确定性高、无法预判所有输入的场景 |

### 2.4 两种形态的关系

```
工作流（Workflow）          智能代理（Agent）
─────────────────          ────────────────────
固定路径                    动态决策
确定性场景                  不确定性场景
可控性强                    灵活度高
实现简单                    适应性强
适合成熟业务                适合探索性场景
```

两者并非互斥，实际系统中常组合使用。例如在本项目中：
- **工具调用环节**（知识库搜索、网络搜索等）以工作流方式编排
- **整体任务决策**（何时搜索、是否输出）由 Agent 自主判断

### 3.6 大模型调研小结

- **推荐文案生成**注重中文表达和创意，通义千问和豆包表现较好
- **知识推理和信息整合**需要逻辑能力，DeepSeek V4 Pro 性价比极其突出（输出仅 ¥6/百万 token）
- **高质量精品内容**适合选用 GPT-5.5 或 Claude 系列
- **成本控制**：国产模型输出价格在 ¥2~¥24/百万 token 区间，远低于国际模型的 ¥100+/百万 token

## 四、搜索方案调研

### 4.1 需求分析

系统需要两种搜索能力：
1. **内部知识库搜索**：在已入库的主播资料中进行语义检索
2. **网络实时搜索**：获取主播最新动态和外部评价

### 4.2 知识库搜索方案

#### 调研范围

覆盖当前主流的开源搜索引擎和向量数据库，重点评估在主播知识库场景下的适用性（2026年数据）。

| 方案 | 类型 | 开源协议 | 部署方式 | 运维成本 | 语义搜索 | 适用规模 |
|-----|------|---------|---------|---------|---------|---------|
| **ChromaDB** | 向量数据库 | Apache 2.0 | 嵌入/独立服务 | ⭐ 低 | ✅ 原生 | 小-中型 |
| **Elasticsearch** | 搜索引擎 | Elastic License | 独立集群 | ⭐⭐⭐⭐⭐ 高 | ✅ 需配置 | 大型 |
| **Meilisearch** | 搜索引擎 | MIT | 独立服务 | ⭐⭐ 中低 | ⚠️ 实验性 | 中型 |
| **Typesense** | 搜索引擎 | GPL-3.0 | 独立服务 | ⭐⭐ 中低 | ✅ 内置 | 中型 |
| **Milvus** | 向量数据库 | Apache 2.0 | 独立集群 | ⭐⭐⭐⭐ 高 | ✅ 原生 | 大-巨型 |
| **FAISS** | 向量计算库 | MIT | 库引用 | ⭐ 低 | ✅ 需自行封装 | 任意 |

#### 各方案简要分析

**ChromaDB**
- 轻量级，可嵌入应用进程运行，适合小到中型规模
- 原生支持向量相似度搜索和 MMR 搜索
- Python 生态集成友好，与 LangChain 深度适配
- 无独立服务运维开销，适合知识库规模在百万级文档以内的场景

**Elasticsearch**
- 功能最丰富的企业级搜索引擎，支持全文搜索 + 向量搜索
- 分布式架构，水平扩展能力强
- 运维复杂度高，需要专业团队维护
- 资源消耗大，小规模场景下过度设计

**Meilisearch / Typesense**
- 轻量级全文搜索引擎，部署简单
- 适合站内搜索、文档检索场景
- 语义搜索（向量）支持仍在完善中
- 中文分词需额外配置或预处理

**Milvus**
- 专业向量数据库，适合大规模向量检索
- 分布式架构，支持十亿级向量规模
- 部署和运维成本较高，需要独立集群

#### 选型建议

对于主播知识库场景（规模预期数千至数万份文档），ChromaDB 在**功能满足度、部署成本和运维复杂度**之间取得了较好的平衡，适合作为首选方案。如果未来知识库规模大幅增长，可考虑迁移至 Milvus 或 Elasticsearch。

### 4.3 网络搜索方案

| 方案 | 免费额度 | 付费价格 | 特点 |
|-----|---------|---------|------|
| **Serper** | 2,500 次/月 | $50/50K 次（$1/千次） | API 接口极简，响应快，结构化数据丰富 |
| **Tavily** | 1,000 次/月 | $100/12.5K 次（$8/千次） | 专为 AI Agent 优化，内容预处理 |
| **Brave Search** | 2,000 次/月 | $5/月 起 | 隐私保护，新兴方案，性价比高 |
| **Google Search API** | $200/月免费额度 | $5/千次 | 品牌可靠，结果质量高，价格较高 |

### 4.4 搜索方案调研小结

- **内部知识库**：ChromaDB 在小到中型规模下综合成本最低，嵌入部署无需额外运维
- **网络搜索**：Serper 性价比最优，免费额度充足，API 简洁，适合 AI Agent 集成

## 五、综合分析与推荐

### 5.1 大模型选型推荐

| 场景 | 推荐模型 | 核心考虑 |
|------|---------|---------|
| **日常推荐（性价比优先）** | DeepSeek V4 Flash / 豆包 2.0 Lite / 通义千问 Qwen3.5-Plus | 成本极低（¥1~¥4/百万 token），中文表现好 |
| **精品推荐（质量优先）** | GPT-5.5 或 Claude Sonnet 4.6 | 推理和生成质量领先，适合重要主播的精品文案 |
| **合规敏感场景** | 通义千问 / 豆包 / 星火 | 国内模型，数据不出境，合规风险最低 |
| **开源自部署** | DeepSeek V4 Pro / GLM-5.1 | 可本地部署，无调用费用 |

### 5.2 搜索方案推荐

- **知识库**：ChromaDB（轻量嵌入，适合当前规模）
- **网络搜索**：Serper（性价比高，集成简单）

### 5.3 架构推荐

采用 **ReAct Agent + Workflow 混合架构**：
- 整体决策由 Agent 自主判断（何时搜索、是否输出）
- 具体工具执行以工作流方式编排（搜索→获取结果→返回）

## 六、调研结论

通过对大语言模型、搜索方案和业务流程形态的综合调研，形成以下结论：

1. **架构选型**：ReAct Agent 适合主播推荐这类不确定性高的场景，Workflow 适合工具调用等确定性环节，两者组合使用效果最佳

2. **模型选择**：国产模型在中文场景和性价比上具有明显优势，国际模型在复杂推理方面仍领先。2026 年国产模型价格战已使成本降至国际模型的 1/10~1/30，建议采用"国产模型日常使用 + 高端模型精品场景"的组合策略

3. **搜索方案**：ChromaDB + Serper 的组合在成本、功能和部署复杂度之间取得较好平衡，适合当前项目规模

4. **成本控制**：采用 DeepSeek V4 Flash 或豆包 Lite 等国产轻量模型，百万字符处理成本可控制在 ¥1~¥4，年运营成本极低
